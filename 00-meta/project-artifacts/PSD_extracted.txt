All rights reserved.1. PROBLEM STATEMENTThis project addresses the inherent security vulnerabilities and poor user experience of traditional authentication methods. A cloud-native SaaS (Software as a Service) platform will be developed, integrating multi-biometric data to provide high resistance against fraud for both physical and digital access scenarios. The system aims to maximize security through an innovative liveness detection algorithm and ensure easy integration via developer-friendly APIs.2. PROBLEM DESCRIPTION AND MOTIVATIONWith the proliferation of digital services and smart buildings, authentication processes have become an integral part of our daily lives. However, widely used methods such as passwords, access cards, or single-factor biometric systems harbor various security risks and usability issues. Passwords can be stolen, cards can be cloned, and simple biometric systems can be easily circumvented by spoofing attacks using static photos or videos. This situation creates significant security gaps for both individual users and corporate entities.The primary motivation for this project is to create a secure, flexible, and user-friendly platform that consolidates these fragmented and insecure authentication methods under a single roof. Targeting B2B (Business-to-Business) and B2B2C (Business-to-Business-to-Consumer) markets, the project aims to unify both physical (e.g., opening doors) and digital (e.g., logging into systems) access control for a wide range of applications, from residential complexes to corporate offices. The platform is intended to differentiate itself from existing solutions and fill a significant market gap through its unique liveness detection mechanism against fraud and its high accuracy achieved by fusing multiple biometric modalities.3. MAIN GOAL AND OBJECTIVESThe main goal of the project is to design, develop, and present a prototype of a multi-tenant, cloud-native, multi-biometric identity authentication SaaS platform that offers a modern, secure, and flexible alternative to traditional authentication systems.The specific objectives set to achieve this main goal are as follows:To design and develop a secure and scalable backend infrastructure. This infrastructure will consist of a main service (Identity Core API) responsible for user, tenant, and role management, and a specialized service (Biometric Processor API) for handling computationally intensive biometric operations, in accordance with microservices architecture principles.To develop an innovative and active liveness detection algorithm, the "Biometric Puzzle," which requires the user to complete a randomly generated sequence of actions (e.g., smiling, blinking) to prevent spoofing attacks using static photos, video recordings, or masks.To integrate high-accuracy face recognition and face verification capabilities into the system using proven deep learning models, such as those provided by the DeepFace library, and to evaluate their performance.To develop a user-friendly mobile application for both Android and iOS platforms (using Flutter) that allows end-users to enroll their biometric data and authenticate their identity.To design the necessary multi-tenant data model for the system to operate as a SaaS platform and to ensure that each tenant's data is completely isolated at the PostgreSQL database level.4. RELATED WORKThe Identity and Access Management (IAM) market is a mature field dominated by major players such as Okta, Auth0, and Microsoft Azure Active Directory. These platforms primarily focus on traditional multi-factor authentication (MFA) methods like passwords, OTPs (One-Time Passwords), and push notifications. While they may offer biometric authentication as a feature, it is often limited to device-based, siloed solutions like Apple's Face ID or Android's Fingerprint API. Furthermore, their capabilities for physical access control are generally limited and require third-party hardware integrations.In academia, numerous studies have been conducted on face recognition and liveness detection. Deep learning-based approaches, in particular, have elevated face recognition accuracy beyond human-level performance (Taigman et al., 2014). Research in liveness detection (Face Anti-Spoofing) has concentrated on passive methods such as texture analysis, blink detection, and the use of 3D cameras. However, these passive methods can be inadequate against increasingly sophisticated video and mask-based attacks.Our project distinguishes itself from these commercial and academic works in several key aspects. Firstly, our platform provides a unified identity for managing both digital and physical access, offering a level of flexibility not commonly found in many market-leading IAM solutions. The second and most significant innovation is our active liveness detection algorithm, the "Biometric Puzzle." By requiring the user to perform a random and sequential series of micro-expressions, this method creates a dynamic and far more secure defense layer against spoofing attacks that can easily bypass passive techniques. Finally, designing the project as a cloud-native, microservices-based SaaS platform with developer-friendly APIs aligns it with modern software development practices, setting it apart from monolithic systems dependent on specific hardware or software.5. METHODOLOGY AND TECHNICAL APPROACHThe project will be built upon a microservices architecture, adhering to modern software development principles with independently deployable modules. This approach aims to enhance the system's flexibility, scalability, and maintainability. The high-level architecture and core workflows are detailed below.6.1. High-Level System ArchitectureThe system consists of microservices running behind an API Gateway, each focused on specific responsibilities. The data layer utilizes specialized solutions for relational data and biometric vectors. Asynchronous communication is facilitated through a message queue to increase system resilience.[System Architecture Diagram to be inserted here - Can be generated from Mermaid code]Figure 1: High-Level System Architecture6.2. Technical Approach and Algorithms6.2.1. Backend ServicesThe backend is composed of two main microservices developed with Hexagonal Architecture principles. The Identity Core API (Spring Boot), acting as the system's brain, manages user, role, and tenant administration, security policies, and business workflows, using Spring Data JPA for data persistence. The Biometric Processor API (FastAPI) leverages Python's powerful machine learning ecosystem to handle computationally intensive tasks like face recognition and liveness detection. Communication between the two services will be handled via REST API calls (synchronous) and a Redis-based message queue (asynchronous).6.2.2. Biometric Puzzle (Liveness Detection) AlgorithmLiveness detection will be performed on the mobile application, utilizing the device's own processing power. The flow is as follows:The system generates a random and sequential command sequence from a predefined pool of actions (Smile, Blink Left Eye, Look Right, etc.).The mobile application starts reading the camera stream using OpenCV.Each video frame is sent to the Google MediaPipe library to obtain the real-time coordinates of 468 facial landmarks.Using these coordinates, metrics such as the Eye Aspect Ratio (EAR) and Mouth Aspect Ratio (MAR) are calculated in real-time.When a calculated metric matches the requested command (e.g., a "blink" is considered successful when the EAR value suddenly approaches zero), the puzzle proceeds to the next step.Once all steps are successfully completed within a specific time frame, liveness is confirmed, and the final frame is sent to the backend for identity verification.6.2.3. Face Recognition and Data StorageFor identity verification, the Biometric Processor API will utilize the DeepFace library with the VGG-Face model. During user enrollment, a user's face is converted into a 2622-dimensional vector (embedding) using the DeepFace.represent() function. This vector is stored in the PostgreSQL database in a special vector column using the pgvector extension. During authentication, a new vector is generated in real-time from the liveness-proven face photo, and the cosine distance between this new vector and the stored vector is calculated. If this distance is below a predetermined threshold, the authentication is considered successful.6.3. Performance EvaluationThe system's performance will be evaluated based on both biometric accuracy and system response times. The accuracy of the biometric model will be measured using standard metrics: False Acceptance Rate (FAR) and False Rejection Rate (FRR). The success of the liveness detection algorithm will be assessed by its resilience against known spoofing attempts (e.g., printed photos, video replays) and its completion time for legitimate users. API response times and database query latencies will be measured through load testing to ensure they meet the target performance criteria.6.4.
