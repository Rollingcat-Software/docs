1. Introduction
1.1 Problem Description
Traditional authentication methods present significant security vulnerabilities and poor user experiences in modern digital and physical access scenarios. According to Verizon's 2024 Data Breach Investigations Report [1], 81% of hacking-related breaches involve stolen or weak passwords, while the Identity Theft Resource Center reports a 78% increase in data compromise events [2] between 2022-2023 affecting over 350 million individuals. Simultaneously, physical access cards can be cloned with readily available $50 RFID readers, and passive biometric systems remain vulnerable to spoofing attacks using static photos (success rate: 67% against basic systems), videos, or sophisticated 3D masks.
This project addresses these challenges by developing FIVUCSAS (Face and Identity Verification Using Cloud-based SaaS) - a multi-tenant, cloud-native biometric authentication platform. The system integrates advanced face recognition with an innovative active liveness detection algorithm called "Biometric Puzzle" to provide robust protection against fraud while maintaining excellent user experience through developer-friendly APIs.
The platform targets B2B and B2B2C markets, unifying both physical access control (door entry, kiosk authentication) and digital authentication (system login, transaction verification) under a single identity management solution.
1.2 Scope
1.2.1 In Scope
Backend Services: The system utilizes a microservices architecture comprising the Identity Core API developed with Spring Boot 3.2 (Java 21) for business logic and the Biometric Processor API built on FastAPI (Python 3.11) for AI-driven operations.
Client Applications: The platform supports multiple interfaces including a Web Admin Dashboard (React 18), a native Mobile App (Android), and a cross-platform Desktop App (Windows/Linux/macOS) developed using Kotlin Multiplatform.
Database Architecture: Data persistence is managed via a multi-tenant PostgreSQL 16 database, utilizing the pgvector extension to efficiently store and query high-dimensional face embeddings [3].
Face Biometrics: The system performs Face Recognition (1:N) and Face Verification (1:1) tasks utilizing the DeepFace [4] library, offering flexibility with 9 different model options.
Liveness Detection: A hybrid approach combines the "Biometric Puzzle" active liveness challenge with passive anti-spoofing techniques, powered by MediaPipe [5] to track 468 facial landmarks.
Infrastructure: Deployment is orchestrated using Docker Compose for containerization, with a NGINX [12] API Gateway handling request routing and Redis [6] employed for high-performance caching.
Security: The platform enforces strict security measures including JWT-based authentication, Role-Based Access Control (RBAC), BCrypt for password hashing, and API rate limiting.
1.2.2 Out of Scope
Additional Biometric Modalities: Although the system's hexagonal architecture [7] is designed for extensibility, the integration of fingerprint, voice, or iris recognition is deferred. This decision allows the team to concentrate on perfecting facial recognition and liveness detection algorithms, avoiding the deployment complexity associated with specialized hardware peripherals like fingerprint scanners.
Production-Grade Orchestration: While the system is cloud-native, complex container orchestration via Kubernetes and Helm charts is excluded from the current scope. Docker Compose [8] is utilized for all development and demonstration purposes, providing a sufficient environment for validating the microservices architecture without the operational overhead of a full production cluster.
Hardware Manufacturing: The project focuses strictly on software engineering principles; therefore, the design and manufacturing of physical access control hardware are excluded. Edge device interactions are simulated through software interfaces, though the system remains compatible with standard commercial hardware such as Raspberry Pi-based units.
Comprehensive Billing Integration: To maintain focus on core biometric innovations, advanced subscription management and third-party payment gateways (e.g., Stripe) are not implemented. Multi-tenancy is enforced through database-level quotas, which is sufficient to demonstrate the platform's SaaS capabilities without implementing generic billing logic.
1.2.3 Constraints
Technology: The project exclusively utilizes open-source technologies with permissive licenses, strictly excluding proprietary databases, machine learning models, or commercial frameworks.
Infrastructure: Development is primarily conducted on local environments utilizing Docker Compose [8]. VPS hosting is permitted solely for demonstration and testing purposes, not for production deployment.
Hardware: Liveness detection accuracy is constrained by the client device's hardware, specifically requiring a recommended minimum camera resolution of 720p. System inference is optimized for CPU execution, with GPU acceleration considered optional for scalability.
Timeline: The project scope is bound by the academic semester constraints (September 2025 - January 2026), necessitating the prioritization of core biometric features over auxiliary functionalities.

2. Literature Survey
This section provides a comprehensive review of academic and industrial research related to biometric authentication systems, with a particular focus on Identity and Access Management (IAM), deep learning-based face recognition, liveness detection, identity document verification, and cloud-native multi-tenant architectures. The objective is to identify limitations in existing approaches and to position the proposed system within the broader biometric and IAM landscape.
2.1 Identity and Access Management (IAM) Systems
Identity and Access Management (IAM) systems form the foundation of secure digital services by regulating authentication, authorization, and user lifecycle management. Leading commercial IAM platforms such as Okta, Auth0, and Microsoft Azure Active Directory have established mature ecosystems centered on traditional authentication paradigms. These platforms primarily employ password-based authentication, multi-factor authentication (MFA) using One-Time Passwords (OTP), push-based approval mechanisms, and device-bound biometric authentication such as Apple Face ID and Android fingerprint recognition.
Despite their widespread adoption, existing IAM solutions exhibit several structural limitations. First, device-bound biometric mechanisms restrict identity portability by tightly coupling authentication to specific hardware ecosystems, resulting in fragmented user experiences. Second, physical access control systems such as doors, turnstiles, and kiosks are typically managed through isolated third-party integrations, leading to disconnected physical and digital identity domains. Finally, many IAM platforms treat biometrics as a supplementary authentication factor rather than a core identity primitive, limiting their resistance to advanced spoofing and replay attacks.
Recent academic and industrial studies emphasize the growing need for IAM systems that unify physical and digital identity under a single trust framework, support strict tenant isolation in shared environments, and integrate advanced biometric verification pipelines. These requirements motivate the adoption of cloud-native, biometric-first IAM architectures capable of scaling across organizations while preserving security, privacy, and regulatory compliance.
2.2 Deep Learning Based Face Recognition Technologies
Face recognition has undergone a significant transformation with the adoption of deep learning-based representation learning. Early face recognition systems relied on handcrafted features such as Local Binary Patterns (LBP) and Histogram of Oriented Gradients (HOG), which were highly sensitive to variations in pose, illumination, and occlusion. The introduction of deep convolutional neural networks (CNNs) enabled end-to-end learning of discriminative facial embeddings, dramatically improving robustness and accuracy.
Pioneering work such as DeepFace demonstrated that deep neural networks could approach human-level performance in face verification by learning high-dimensional facial representations. FaceNet [11] further advanced the field by introducing a unified embedding space optimized using triplet loss, enabling direct comparison of facial identities through simple distance metrics.
Subsequent research focused on enhancing inter-class separability and intra-class compactness through margin-based loss functions. ArcFace [10] introduced additive angular margin loss, achieving state-of-the-art performance on benchmarks such as LFW. Extensions including CosFace, SphereFace, MagFace, and AdaFace incorporated cosine margins, angular normalization, and quality-aware weighting strategies to improve robustness under low-resolution imaging, motion blur, and off-angle facial captures.
While these models achieve exceptional accuracy in controlled benchmark evaluations, most studies focus primarily on algorithmic performance. System-level concerns such as scalability, latency, multi-tenant isolation, and operational monitoring are often outside the scope of academic research. As a result, a gap remains between research-grade face recognition models and production-ready biometric platforms.
To bridge this gap, unified frameworks such as DeepFace [18] (Serengil & Ozpinar) provide standardized access to multiple state-of-the-art face recognition models, abstracting face detection, alignment, embedding generation, and similarity computation. Such frameworks enable the practical deployment of advanced face recognition models in real-world systems while preserving architectural flexibility.

2.3 Liveness Detection and Anti-Spoofing
Liveness detection has become a critical component of biometric authentication systems due to the increasing prevalence of presentation attacks, including printed photographs, replayed videos, and high-quality deepfake content. Face anti-spoofing techniques are commonly categorized into passive, active, and hybrid approaches , as standardized in ISO/IEC 30107-3 [9].
Passive liveness detection methods analyze visual cues from single images or video streams without requiring user interaction. Common techniques include texture analysis using Local Binary Patterns, frequency-domain analysis for detecting moiré artifacts, color space consistency checks, and CNN-based classifiers trained on spoofing datasets. Although passive approaches offer a seamless user experience, they remain vulnerable to high-quality attacks, three-dimensional masks, and generative AI-based spoofing.
Active liveness detection introduces challenge-response mechanisms that require deliberate user actions, such as blinking, smiling, head movements, or gaze tracking. Quantitative metrics such as Eye Aspect Ratio (EAR) and Mouth Aspect Ratio (MAR) are widely used to detect these actions. The emergence of real-time facial landmark frameworks such as Google MediaPipe Face Mesh, which provides 468 facial landmarks, has significantly improved the feasibility of on-device active liveness verification.
Hybrid approaches combine passive and active techniques to maximize robustness against both traditional and AI-generated attacks. Recent literature highlights that hybrid pipelines achieve superior security while maintaining acceptable latency and usability. However, many proposed solutions remain model-centric and lack integration into complete, end-to-end authentication workflows.
The increasing sophistication of generative adversarial attacks underscores the need for liveness mechanisms that are dynamic, randomized, and context-aware-capabilities that static classifiers alone cannot reliably provide.
2.4 Identity Document Verification and Standards
In high-assurance identity verification scenarios, biometric authentication is frequently complemented by identity document verification. Research in this area includes ID-selfie matching pipelines, optical character recognition (OCR), Machine Readable Zone (MRZ) parsing, and document layout analysis. Systems such as DocFace demonstrate that ID-specific face matching models outperform generic face recognition approaches when comparing document photographs with live facial captures.
International standards play a crucial role in document-based identity verification. The International Civil Aviation Organization (ICAO) defines specifications for electronic Machine-Readable Travel Documents (eMRTDs), including secure storage of facial biometric data in NFC-enabled passport chips. NFC-based verification workflows provide strong cryptographic guarantees and are widely adopted in border control and e-government systems.
Commercial solutions such as BioGATE Pass leverage ICAO-compliant NFC pipelines to achieve high levels of trust. However, these systems are often tightly coupled to specialized hardware and lack extensibility for broader cloud-based SaaS deployments. Consequently, recent academic literature increasingly emphasizes modular and extensible architectures that allow document verification components to coexist with biometric and behavioral authentication modules.
2.5 Industrial Biometric Authentication Solutions
Several industrial platforms provide large-scale face recognition and liveness detection services. Microsoft Azure Face Liveness and Amazon Rekognition offer highly accurate, standards-compliant biometric APIs with global availability. However, these platforms are fully proprietary, cloud-dependent, and provide limited transparency, customization, and control over data processing. Privacy-sensitive organizations and academic institutions often face regulatory, cost, and vendor lock-in concerns when adopting such services.
Regional providers such as Sodec Technologies focus primarily on e-government and banking applications but typically lack multi-tenant SaaS architectures and advanced active liveness detection mechanisms. Open-source solutions such as FaceAuth Vision support offline and edge deployments but fall short in modern anti-spoofing robustness and enterprise-grade scalability.
Comparative analyses indicate that no single existing solution simultaneously provides open extensibility, hybrid liveness detection, cloud-native scalability, and unified management of physical and digital identities.
2.6 Cloud-Native Biometric Systems and Multi-Tenant Architectures
Recent research emphasizes the importance of cloud-native architectural principles such as microservices, containerization, and event-driven communication for scaling biometric workloads. Vector databases such as pgvector [3] and similarity search libraries like FAISS [13] enable efficient storage and retrieval of high-dimensional facial embeddings, supporting large-scale identification and verification scenarios.
Multi-tenant architectures introduce additional challenges, including strict data isolation, tenant-specific configuration, and performance fairness under shared infrastructure. While numerous studies address individual aspects of biometric scalability or tenant isolation, relatively few systems integrate identity management, biometric processing, and liveness detection within a single cohesive architectural framework.
Modern architectural patterns such as Hexagonal Architecture and Clean Architecture are increasingly recommended to decouple core business logic from infrastructure concerns, enabling maintainability, testability, and long-term system evolution.

2.7 Positioning and Differentiation of the Proposed System
The reviewed literature reveals a clear gap between state-of-the-art biometric research and deployable, multi-tenant IAM platforms. Existing solutions either focus narrowly on algorithmic innovation or offer closed, inflexible commercial systems with limited extensibility.
The proposed system addresses this gap by integrating advanced deep learning-based face recognition models, hybrid liveness detection, and identity document verification within a cloud-native SaaS architecture. By combining active challenge-response liveness mechanisms with configurable multi-model face recognition pipelines, the platform achieves strong resistance to spoofing while maintaining usability and scalability. Its microservices-based design enables unified management of physical and digital access, tenant-level customization, and extensibility toward future biometric modalities.
In this context, the system positions itself as a bridge between academic biometric research and practical, enterprise-grade identity platforms, aligning with contemporary trends in secure, scalable, and privacy-aware authentication systems.

3.1.1 Identity and Access Management (FR-1)
3.1.1.1 Description
The system shall provide a secure mechanism for authenticating users and managing administrative accounts within strictly isolated tenant boundaries.
3.1.1.2 Inputs
* User credentials (email address, password)
* Tenant identifiers
* Session-related information (IP address, User-Agent, timestamp)
3.1.1.3 Processing
1. Validate credential formats (RFC 5322 for email, minimum password requirements)
2. Enforce user uniqueness within each tenant namespace
3. Verify password against stored BCrypt hash (work factor 12)
4. Generate signed JWT access token (15-minute expiry) and UUID-based refresh token (7-day expiry)
5. Store session metadata in Redis cache
3.1.1.4 Outputs
* Authentication tokens (Access Token, Refresh Token)
* User profile information (ID, email, roles, tenant context)
* Session expiration timestamps
3.1.1.5 Error/Data Handling
* Invalid Credentials: HTTP 401 Unauthorized with generic error message
* Duplicate Account: HTTP 409 Conflict during registration
* Locked Account: HTTP 423 Locked after 5 consecutive failed attempts
* Invalid Token: HTTP 401 Unauthorized for expired or malformed tokens

3.1.2 Biometric Enrollment (FR-2)
3.1.2.1 Description
The system shall support the secure enrollment of facial biometric data after verifying both image quality and user liveness.
3.1.2.2 Inputs
* Facial images captured during interactive liveness challenge
* User identifier (from authenticated session)
* Tenant context
3.1.2.3 Processing
1. Detect facial region using configurable backend (RetinaFace, MTCNN, or MediaPipe)
2. Assess biometric sample quality (brightness > 0.3, sharpness > 0.5, frontal pose within 15°)
3. Validate liveness through 'Biometric Puzzle' challenge sequence
4. Generate standardized biometric embedding using selected model (default: Facenet512)
5. Store embedding vector in PostgreSQL with pgvector extension
3.1.2.4 Outputs
* Unique enrollment identifier (UUID)
* Quality confidence score (0.0 - 1.0)
* Enrollment status (SUCCESS, PENDING, FAILED)
3.1.2.5 Error/Data Handling
* No Face Detected: HTTP 400 Bad Request with guidance message
* Image Quality Insufficient: HTTP 422 Unprocessable Entity with specific quality metrics
* Liveness Validation Failed: HTTP 403 Forbidden with failure reason (e.g., 'Blink not detected')

3.1.3 Biometric Verification (FR-3)
3.1.3.1 Description
The system shall verify user identity by comparing live facial biometric samples against previously enrolled biometric templates, with optional liveness validation.
3.1.3.2 Inputs
* Live facial image (base64 encoded or multipart upload)
* User or tenant context for template retrieval
* Optional: Liveness challenge response
3.1.3.3 Processing
1. Generate biometric embedding from input image using configured model
2. Retrieve enrolled reference template(s) from vector database
3. Compute cosine similarity score between input and reference embeddings
4. Compare similarity score against configurable threshold (default: 0.6)
5. Log verification attempt with result and metadata
3.1.3.4 Outputs
* Verification decision (MATCH / NO_MATCH)
* Similarity confidence score (0.0 - 1.0)
* Matched identity information (if successful)
3.1.3.5 Error/Data Handling
* No Enrollment Found: HTTP 404 Not Found
* Verification Failed: HTTP 401 Unauthorized when score < threshold
* Rate Limit Exceeded: HTTP 429 Too Many Requests

3.1.4 Multi-Tenant Management (FR-4)
3.1.4.1 Description
The system shall support isolated configurations for multiple tenants, enabling independent management of quotas, limits, and operational settings.
3.1.4.2 Inputs
* Tenant metadata (organization name, domain, contact email)
* Configuration parameters (max users, enrollment quota, API rate limits)
* Billing tier selection
3.1.4.3 Processing
1. Validate tenant identifier uniqueness across the platform
2. Provision tenant-specific configuration with default quotas
3. Generate tenant API credentials (API Key, Secret)
4. Apply row-level security policies for data isolation
3.1.4.4 Outputs
* Unique tenant identifier (UUID)
* Tenant API credentials
* Configuration confirmation with applied quotas
3.1.4.5 Error/Data Handling
* Duplicate Tenant: HTTP 409 Conflict
* Quota Exceeded: HTTP 429 Too Many Requests when tenant limits are reached
* Invalid Configuration: HTTP 422 Unprocessable Entity for invalid parameters

3.1.5 Authorization & Role Based Access Control (FR-5)
3.1.5.1 Description
The system shall enforce role-based access control to regulate access to protected resources based on assigned permissions.
3.1.5.2 Inputs
* Requesting user identity (extracted from JWT)
* Target resource identifier (URI path)
* Requested operation (HTTP method: GET, POST, PUT, DELETE)
3.1.5.3 Processing
1. Decode and validate JWT token signature and expiration
2. Extract user roles from token claims
3. Resolve role-to-permission mappings from RBAC configuration
4. Evaluate if user permissions include the requested operation on the target resource
5. Cache permission decisions for performance (TTL: 5 minutes)
3.1.5.4 Outputs
* Authorization decision (ALLOW / DENY)
* Effective permissions list (for debugging/audit)
3.1.5.5 Error/Data Handling
* Insufficient Permissions: HTTP 403 Forbidden
* Invalid Token: HTTP 401 Unauthorized
* Resource Not Found: HTTP 404 Not Found

3.1.6 Auditing & Compliance (FR-6)
3.1.6.1 Description
The system shall record and expose immutable audit logs and verification histories to support security monitoring and compliance with applicable data protection regulations (GDPR, KVKK).
3.1.6.2 Inputs
* User action details (actor, action type, target resource)
* Request context (IP address, User-Agent, timestamp)
* Affected data identifiers
3.1.6.3 Processing

2. Append ISO 8601 timestamp and tenant context
3. Generate immutable audit record with unique event ID
4. Write to append-only audit log storage with tenant isolation
3.1.6.4 Outputs
* Paginated audit record list with filtering support
* Verification history reports (exportable as CSV/JSON)
* Compliance summary dashboards
3.1.6.5 Error/Data Handling

The following requirements specify the operational criteria and constraints that define the system's quality and performance. These are categorized by their specific attributes to ensure the platform is robust, secure, and maintainable.

3.2.1 Performance
The system shall provide responsive system behavior for authentication and biometric operations in order to support real-time usage scenarios such as physical access control, kiosk authentication, and online identity verification. Performance requirements are critical due to the computational nature of biometric processing and the user-facing interaction model of the system. The platform shall ensure that authentication, enrollment, and verification workflows complete within acceptable time bounds so that biometric verification does not introduce perceptible delays or degrade user experience. Performance compliance shall be evaluated through controlled testing under representative workloads.
3.2.2 Scalability
The system shall be designed to scale horizontally in order to support multiple tenants, increasing user populations, and growing biometric workloads. As a Software-as-a-Service platform targeting B2B and B2B2C use cases, the system must accommodate variations in tenant size and request volume without requiring architectural redesign. Scalability requirements ensure that biometric enrollment and verification services can handle concurrent requests across tenants while preserving isolation and performance. Scalability shall be validated through load and stress testing aligned with projected usage scenarios.
3.2.3 Reliability
The system shall operate reliably during designated operational periods, ensuring consistent availability of identity verification services. Reliability is a critical requirement due to the system's role in security-sensitive workflows, where service interruptions may block user access or disrupt operations. The system shall tolerate partial failures and recover gracefully without cascading effects across components or tenants. Reliability targets shall be assessed through availability monitoring, fault injection, and controlled failure scenarios.
3.2.4 Security
The system shall enforce strict security controls to protect identity information and biometric data throughout their lifecycle. Security requirements address confidentiality, integrity, and access control for sensitive data, particularly given the irreversible nature of biometric identifiers. The system shall prevent unauthorized access, ensure strong isolation between tenants, and minimize exposure of biometric data by operating on derived representations rather than raw biometric artifacts. Compliance with security requirements shall be verified through security testing, auditability, and adherence to best practices in secure system design.
3.2.5 Usability
The system shall provide an intuitive and efficient user experience across all supported client applications, including web, mobile, and desktop environments. Usability requirements are especially important for biometric interactions, where unclear feedback or complex flows can lead to user frustration or enrollment failure. The system shall guide users through enrollment and verification processes with clear instructions and meaningful feedback, enabling successful completion with minimal effort. Usability effectiveness shall be evaluated based on task completion success and consistency of interaction flows.
3.2.6 Maintainability
The system shall be maintainable and extensible to support future enhancements, algorithm upgrades, and evolving security requirements. Given the rapid advancement of biometric technologies and identity verification standards, the system must allow components to be modified or replaced without disrupting core functionality. Maintainability requirements emphasize modular structure, clear separation of concerns, and ease of testing, enabling the system to evolve throughout its lifecycle. Compliance with maintainability goals shall be assessed through architectural conformance and codebase organization.
3.2.7 Portability
The system shall operate consistently across supported deployment environments and client platforms. Portability requirements ensure that the system can be deployed in different infrastructure settings and accessed through multiple operating systems without behavioral inconsistencies. This is essential for a SaaS platform intended for integration into diverse organizational environments. Portability shall be validated through deployment and execution testing across supported platforms and environments.

5. Software Architecture
The software architecture of FIVUCSAS is designed as a scalable, cloud-native ecosystem that prioritizes modularity, high-performance biometric processing, and strict multi-tenant isolation. By adopting a microservices-based approach, the system decouples administrative identity management from compute-intensive machine learning tasks, ensuring that each component can scale independently based on demand. The architecture is organized into distinct logical layers-ranging from cross-platform client interfaces to a robust data layer optimized for vector similarity search all orchestrated through a centralized API Gateway that ensures secure and efficient request routing across the entire platform.[7][16]
5.1 Architectural Style
FIVUCSAS employs a microservices architecture with Hexagonal Architecture (Ports & Adapters) within each service, ensuring:
* Separation of Concerns: Business logic isolated from infrastructure
* Testability: Domain logic testable without external dependencies
* Flexibility: Easy to swap implementations (e.g., database, ML models)
* Scalability: Services scale independently based on load [7]

Within FIVUCSAS, each microservice (e.g., Identity Core API and Biometric Processor API) independently applies the Hexagonal Architecture, where external interfaces such as REST APIs and message consumers interact with application use cases through defined ports. This approach ensures that core domain logic remains independent of frameworks, databases, and communication mechanisms. [7]

6. Tasks Accomplished
6.1 Current State of the System
FIVUCSAS platform provides an integrated facial analysis framework that combines modern biometric techniques with a modular and scalable software architecture. The system brings together face detection, face recognition, liveness detection, and quality assessment components into a unified processing pipeline, establishing a solid experimental foundation for secure identity verification applications.
The facial analysis pipeline begins with a face detection module supporting robust multi-face detection through configurable backend engines. For each detected face, precise bounding box extraction is performed, followed by face alignment and normalization as illustrated in Figure 13. These preprocessing steps aim to ensure consistency across varying acquisition conditions and to improve the reliability of downstream biometric processing.

Figure 13: Multi-face detection, landmark extraction, and real-time identity matching results
The face recognition component processes normalized facial images to generate high-dimensional biometric embeddings. These embeddings are stored in a multi-tenant PostgreSQL database enhanced with pgvector, enabling efficient similarity search and identity comparison. The subsystem supports embedding comparison as well as batch processing, allowing the system to be evaluated under both real-time and higher-volume identification scenarios.
To address spoofing and presentation attacks, the platform incorporates a multi-layer liveness detection strategy that combines active and passive approaches. The active liveness mechanism follows a Biometric Puzzle concept implemented using MediaPipe, where users are required to complete randomized challenge sequences. These challenges include blink detection based on the Eye Aspect Ratio (EAR), smile detection using the Mouth Aspect Ratio (MAR), and head pose tracking across yaw, pitch, and roll axes. This challenge-response structure is designed to reduce susceptibility to replay-based attacks. Complementing this, the passive liveness layer analyzes intrinsic image characteristics without explicit user interaction (Figure 14), employing texture analysis via Local Binary Patterns (LBP), color distribution analysis, moiré pattern detection, and frequency-domain analysis to identify presentation artifacts.
In parallel, a dedicated quality assessment module evaluates facial samples prior to biometric processing. This module assesses brightness levels, sharpness and blur metrics, facial pose compliance, occlusion presence, and minimum resolution requirements, ensuring that unsuitable samples are filtered out before reaching subsequent processing stages.




Figure 14: Presentation attack scenario using printed facial images and passive liveness analysis
Beyond facial biometrics, the system includes a specialized NFC integration module capable of authenticating and extracting data from multiple card types as demonstrated in Figure 15, including Turkish National Identity Cards, ICAO 9303-compliant biometric passports, driver's licenses, Marmara University student identification cards, and academic staff cards. This integration enables the exploration of combined physical token-based authentication and digital biometric verification workflows.

Figure 15: Passport detection and classification within the NFC-enabled card authentication module
The overall system is supported by a production-oriented backend infrastructure, exposing more than 46 RESTful endpoints documented via Swagger/OpenAPI. The architecture follows Hexagonal Architecture principles, providing a clear separation between core biometric logic and infrastructure components, and facilitating maintainability, extensibility, and iterative development.
In its current state, the FIVUCSAS platform represents a stable and functional experimental system in which the core biometric pipeline and supporting services have been implemented and validated in controlled environments. Ongoing efforts focus on extended field testing, performance evaluation, and robustness analysis under real-world conditions, guiding the continued evolution of the system in subsequent development phases.
